= Understanding FOMOD Installers

This page explains what FOMOD installers are, how they work, and how all the individual pieces fit together.
It is written for complete beginners - no prior knowledge is required.

== What is a FOMOD installer?

When you download a mod for games like _Skyrim_ or _Fallout 4_, you sometimes get a step-by-step setup wizard in your mod manager - similar to a regular software installer on Windows.
That wizard is a **FOMOD installer**.

It lets the mod author present the user with choices:

* "Do you want the high-resolution textures or the performance-friendly version?"
* "Which other mods do you have installed? I'll automatically select compatible patches for you."
* "Do you want to install the optional music pack?"

The mod manager reads the FOMOD installer, displays the choices, and then copies only the selected files to the game's data folder.

== File structure

A FOMOD installer lives in a subfolder named `fomod/` inside the mod's root directory.
It always consists of exactly two XML files:

[source]
----
<mod root>/
└─ fomod/
   ├─ info.xml          <1>
   └─ ModuleConfig.xml  <2>
----
<1> Metadata: name, author, version, website, description.
<2> The actual installer logic: steps, options, files to install.

Both files are plain XML - a simple, human-readable text format.
The FOMOD Creation Kit manages these files for you so you never have to edit raw XML by hand.

== `info.xml` - Mod metadata

The `info.xml` file describes the mod itself.
It contains:

[cols="1,3"]
|===
|Field |Description

|`Name`
|The name of the mod shown in the mod manager.

|`Author`
|The name of the person who made the mod.

|`Version`
|The current version number (e.g. `1.2.3`).

|`Website`
|A link to the mod's page (e.g. on Nexus Mods).

|`Description`
|A short description of the mod.

|`Id`
|An optional numeric ID used by some mod managers to identify the mod.
|===

.Example info.xml
[source,xml]
----
<fomod>
  <Name>My Awesome Mod</Name>
  <Author>YourName</Author>
  <Version>1.0.0</Version>
  <Website>https://www.nexusmods.com/skyrimspecialedition/mods/12345</Website>
  <Description>Adds a new sword to the game.</Description>
</fomod>
----

== `ModuleConfig.xml` - The installer logic

The `ModuleConfig.xml` file defines everything the installation wizard shows and does.
Think of it as a decision tree: each step can ask the user questions, and the answers determine which files get installed.

=== The big picture

[source]
----
ModuleConfig
├─ Module Name & Image (shown at the top of the installer)
├─ Module Dependencies (requirements the whole mod has)
├─ Required Install Files (always installed, no matter what the user picks)
├─ Install Steps (the wizard pages)
│   ├─ Install Step 1
│   │   ├─ Visibility Condition (optional - hide this step unless ...)
│   │   ├─ Group A  <- e.g. "Choose your texture quality"
│   │   │   ├─ Plugin 1  <- "High Resolution"
│   │   │   ├─ Plugin 2  <- "Medium Resolution"
│   │   │   └─ Plugin 3  <- "Performance"
│   │   └─ Group B  <- "Optional extras"
│   │       └─ Plugin 1  <- "Add music pack"
│   └─ Install Step 2
│       └─ ...
└─ Conditional File Installs (files installed based on flag combinations)
----

=== Install steps

An **install step** is one page in the wizard.
A typical installer has one or more steps.
Each step has a name and contains one or more groups.

A step can optionally have a **visibility condition**.
If the condition is not met, the entire step is skipped and the user never sees it.
This is useful for follow-up questions that only make sense depending on earlier answers.

=== Groups

A **group** collects a set of related plugins (options) and enforces a selection rule.
The available rules are:

[cols="1,3"]
|===
|Rule |Meaning

|`SelectAtLeastOne`
|The user must pick at least one option. Multiple selections are allowed.

|`SelectAtMostOne`
|The user may pick one option or none.

|`SelectExactlyOne`
|The user must pick exactly one option.

|`SelectAll`
|All options are always selected. The user cannot deselect them.

|`SelectAny`
|The user may pick any number of options, including none.
|===

=== Plugins (options)

A **plugin** is a single selectable option inside a group - for example "High-Resolution Textures".

Every plugin can have:

[cols="1,3"]
|===
|Part |Description

|`name`
|The label shown in the installer.

|`description`
|Explanatory text shown when the option is highlighted.

|`image`
|A preview image shown next to the description.

|`files`
|The files and folders to install when this option is selected.

|`conditionFlags`
|Flags to set when this option is selected (see xref:#_flags[Flags] below).

|`typeDescriptor`
|Whether this option is optional, required, recommended, etc.
(see xref:#_plugin_types[Plugin types] below).
|===

==== Plugin types

The **type** of a plugin controls how the mod manager presents it:

[cols="1,3"]
|===
|Type |Behaviour

|`Optional`
|Not pre-selected. The user may select it freely.

|`Required`
|Always selected and cannot be deselected by the user.

|`Recommended`
|Pre-selected but can be deselected if the user wishes.

|`NotUsable`
|Greyed out. Used when a prerequisite is missing and selecting this option would break the game.

|`CouldBeUsable`
|Shown as partially available. The prerequisite plugin exists but is not active.
|===

The type can be set **statically** (always the same) or **dynamically** (changes based on what the user has installed - see xref:#_dependencies[Dependencies]).

== Flags

Flags are the communication channel between installer steps.
A flag is a simple key-value pair, for example `Readme=On`.

* A **plugin** can *set* a flag when the user selects it (`conditionFlags`).
* A later **step**, **group**, or **conditional file install** can *read* that flag (`flagDependency`) to decide what to show or install.

.Example flow using flags
====
1. Step 1 contains a "I have read the readme" checkbox.
   When the user ticks it, the plugin sets the flag `Readme=On`.
2. Step 2 has a visibility condition: _only show this step if_ `Readme=On`.
   This means Step 2 is hidden until the user acknowledges the readme.
====

Flags exist only during the installation process and are not stored anywhere permanently.

== Dependencies

A **dependency** is a condition the installer checks against the user's current setup.
There are three kinds:

[cols="1,3"]
|===
|Type |What it checks

|`fileDependency`
|Whether a specific game plugin file (`.esp`, `.esm`, ...) is *Missing*, *Inactive* (installed but disabled), or *Active* (installed and enabled).

|`flagDependency`
|Whether a condition flag has a specific value (see xref:#_flags[Flags] above).

|`gameDependency` / `fommDependency`
|Whether the installed game or mod manager meets a minimum version requirement.
|===

Multiple dependencies can be combined with an **operator**:

* `And` - all conditions must be true.
* `Or` - at least one condition must be true.

Dependencies can be **nested**: an `And` group can contain an `Or` group inside it, allowing complex logical expressions.

=== Dynamic plugin types

Instead of a fixed plugin type, a plugin can use a `dependencyType`.
The installer checks a list of patterns from top to bottom.
The **first pattern whose dependencies are met** determines the plugin type.
If no pattern matches, the `defaultType` is used.

.Example
====
A plugin is `Recommended` if `PatchMod.esp` is active, but `Optional` otherwise:

[source,xml]
----
<typeDescriptor>
  <dependencyType>
    <defaultType name="Optional"/>
    <patterns>
      <pattern>
        <dependencies operator="And">
          <fileDependency file="PatchMod.esp" state="Active"/>
        </dependencies>
        <type name="Recommended"/>
      </pattern>
    </patterns>
  </dependencyType>
</typeDescriptor>
----
====

== Files and folders

Every plugin (and also the required install files section) references a list of files and folders to install.
Each entry specifies:

[cols="1,3"]
|===
|Attribute |Description

|`source`
|The path to the file or folder *inside the FOMOD package*.

|`destination`
|The path where the file or folder is installed *in the game's data directory*.
If omitted, the destination is the same as the source.

|`priority`
|A number that controls the installation order when multiple plugins install files to the same destination.
A **higher** number means the file is installed **later** and therefore wins over files with a lower priority.

|`alwaysInstall`
|If `true`, the file is installed even if the plugin is not selected.

|`installIfUsable`
|If `true`, the file is installed unless the plugin type is `NotUsable`.
|===

== Conditional file installs

The **conditional file installs** section is a special list at the bottom of the `ModuleConfig.xml`.
It contains patterns - each pattern has a set of dependencies and a list of files.
After all install steps are completed, every pattern whose dependencies are satisfied will have its files installed automatically.

This is commonly used to install compatibility patches without asking the user:
the installer simply checks whether the relevant mods are active and installs the matching patch silently.

.Example
====
Install `Patch_Rugnarok.esp` only when the `Rugnarok=On` flag was set earlier:

[source,xml]
----
<conditionalFileInstalls>
  <patterns>
    <pattern>
      <dependencies operator="And">
        <flagDependency flag="Rugnarok" value="On"/>
      </dependencies>
      <files>
        <file source="patches/Patch_Rugnarok.esp"
              destination="Patch_Rugnarok.esp"/>
      </files>
    </pattern>
  </patterns>
</conditionalFileInstalls>
----
====

== Module dependencies

`moduleDependencies` can be placed at the top of the `ModuleConfig.xml` to define requirements for the *entire* mod.
If the user's setup does not satisfy these dependencies, the mod manager may refuse to install the mod at all.

== Putting it all together

The diagram below shows a minimal but complete FOMOD installer as an example:

[source]
----
Install Step: "Choose your variant"
│
├─ Group: "Quality" (SelectExactlyOne)
│   ├─ Plugin: "Ultra"         → installs textures/ultra/
│   ├─ Plugin: "High"          → installs textures/high/
│   └─ Plugin: "Performance"   → installs textures/performance/
│
└─ Group: "Music Pack" (SelectAny)
    └─ Plugin: "Add music"      → installs music/

Required Install Files: core/MyMod.esp  (always installed)
----

No matter which texture quality the user picks, `MyMod.esp` is always installed.
The music pack is optional and completely independent.
